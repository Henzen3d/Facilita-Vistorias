# Phase 3.1: Produtividade em Campo — Context

**Gathered:** 2026-07-17  
**Status:** Ready for planning / awaiting approval  
**Depends on:** Phase 2 (field PWA offline) + Phase 3 (IA/revisão/PDF — não bloquear UI de captura)

## Phase Boundary

Aumentar a velocidade do vistoriador em campo **sem** implementar Phase 4 (WhatsApp/contestação), medidores, assinatura digital, OCR de contrato, canetinha em foto ou Edge AI.

**In scope (3 entregas):**

1. **Templates** de imóvel → ambientes + itens padrão (offline-first no field)
2. **Navegação “próximo item”** após salvar captura
3. **Score de completude acionável** no resumo (lista “o que falta” com deep links)

**Out of scope (ondas seguintes):**

- Medidores água/luz/gás
- Finalize PDF no field / trava pós-FINALIZADA
- WhatsApp do relatório, galeria pública, contestação
- Modo Rápido vs Detalhe formal (pode ser esboçado depois)
- Reprocessar IA (UI)
- Anotação em foto, GPS, comparação entrada×saída

## Product Decisions (proposed defaults)

| ID | Decision | Rationale |
|----|----------|-----------|
| D-01 | Templates são **catálogo estático no cliente** (`src/lib/field/templates.ts`), não CRUD admin no MVP desta onda | Offline, zero schema Prisma novo, entrega rápida |
| D-02 | Aplicar template só se a vistoria tiver **0 ambientes** (ou confirmação se já houver) | Evita duplicar cômodos por toque acidental |
| D-03 | Ao adicionar **um ambiente avulso**, oferecer itens padrão daquele tipo de cômodo | Produtividade mesmo sem template de imóvel |
| D-04 | Após salvar item: ir para o **próximo PENDENTE do mesmo ambiente**; se não houver, próximo PENDENTE da vistoria; se não houver, lista do ambiente | Menos “voltar → abrir → voltar” |
| D-05 | **Foto obrigatória por item** para considerar o item apto a fechar captura e para habilitar “Finalizar” no resumo. Score trata item sem foto como **blocker** (não só aviso). Áudio continua opcional no MVP desta onda (IA fica mais rica com áudio, mas o PDF exige evidência visual) | Relatório fotográfico precisa ilustrar o que está descrito; item sem imagem não serve no laudo comercial/jurídico da operação |
| D-06 | Itens criados por template entram com `status: PENDENTE` e sem mídia | Igual criação manual atual |
| D-07 | Mutações locais usam ações já previstas (`CREATE_AMBIENTE_LOCAL` / `CREATE_ITEM_LOCAL` ou payload tipado consistente) e sync via PUT existente quando online | Não inventar backend de templates nesta onda |
| D-08 | Linguagem UI: “relatório” / “cômodos” / “itens” — nunca “laudo” | CREA / PRD |

## Success Criteria (what must be TRUE)

1. Vistoriador cria vistoria local (ou abre agendada sem cômodos) e em **≤ 2 toques** aplica um template (ex.: Apto 2 quartos) gerando ambientes + itens no IndexedDB.
2. Após salvar um item, o app abre automaticamente o **próximo item pendente** (sem voltar à lista).
3. Tela de resumo mostra **% de completude** e lista de pendências (sem ambientes, itens sem status, **itens sem foto**, protocolo incompleto, sync pendente), cada linha clicável para o ponto certo. **Finalizar fica desabilitado** se existir item sem foto ou ainda PENDENTE.
4. Tudo funciona **offline** (templates e score só leem/escrevem IDB).
5. Criação manual de ambiente/item continua disponível.

## Risks

- **Sync de CREATE_AMBIENTE/CREATE_ITEM:** hoje a UI enfileira payloads mistos (`UPDATE_VISTORIA_STATUS` + `tipo: CREATE_AMBIENTE`). Templates vão criar muitos registros de uma vez — Plan 01 deve **normalizar mutações** e garantir que o PUT do servidor aceite bulk ou N mutations, sem bloquear o field offline.
- **Vistorias pré-carregadas do admin** que já vêm com ambientes do seed: template não deve sobrescrever sem confirmação (D-02).
- **IDs locais** (`amb-local-*`, `item-local-*`) vs UUIDs do servidor: manter padrão atual da criação manual.
