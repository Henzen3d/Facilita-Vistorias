# Plano de Implementação — Fase 03.1: Produtividade em Campo

**Goal:** Reduzir cliques do vistoriador em campo com templates de cômodos/itens, navegação automática para o próximo item e score de completude acionável no resumo.

**Não é oneshot:** 3 plans sequenciais (Waves). Cada plan fecha com critério de aceite testável no celular. **Não implementar medidores, Phase 4, IA reprocess, canetinha.**

**Referências:** `03.1-CONTEXT.md` (D-01…D-08), análise competitiva Flash × Facilita (2026-07-17).

---

## Escopo congelado

| Inclui | Exclui (próximas ondas) |
|--------|-------------------------|
| Catálogo de templates estático | CRUD admin de templates |
| Aplicar template de imóvel | Medidores |
| Itens padrão ao criar cômodo avulso | Finalize PDF no field |
| Próximo item automático na captura | WhatsApp / contestação |
| Score + lista de gaps no resumo | OCR contrato, assinatura, edge AI |
| Normalização mínima de mutações create ambiente/item | Refator total do sync engine |

---

## Ondas de execução

```text
Wave A (Plan 01)  Templates + apply offline
       │
       ▼
Wave B (Plan 02)  Navegação próximo item + UX lista ambiente
       │
       ▼
Wave C (Plan 03)  Score acionável no resumo
```

Dependência: B depende de A (precisa de muitos itens para valer a navegação).  
C depende de A (score “sem ambientes / itens vazios” faz sentido com templates). B e C são quase paralelos após A; ordem recomendada **A → B → C**.

## Plan 01 — Templates de imóvel / ambiente / itens

### Objetivo

Em vistoria com zero ambientes, o vistoriador aplica um template e passa a ter cômodos + itens padrão no IndexedDB, prontos para capturar.

### Tasks

#### T-01: Catálogo estático de templates

**NEW** `src/lib/field/templates.ts`

- Exportar tipos:
  - `RoomTemplate { nome, itens: string[] }`
  - `PropertyTemplate { id, label, description?, rooms: RoomTemplate[] }`
- Catálogo inicial (ajustável, PT-BR):

| id | label | Cômodos (resumo) |
|----|-------|------------------|
| `studio` | Studio / kitnet | Sala/quarto integrado, banheiro, cozinha, área serviço (se couber em 1 bloco cozinha+serviço) |
| `apto-1` | Apartamento 1 quarto | Sala, cozinha, banheiro, quarto, área de serviço, sacada (opcional no template) |
| `apto-2` | Apartamento 2 quartos | + 2º quarto |
| `apto-3` | Apartamento 3 quartos | + 3º quarto |
| `casa` | Casa | Sala, cozinha, 2 banheiros, 2–3 quartos, área externa/quintal, garagem (simplificado) |

#### T-02: Helper de aplicação (IDB)

**NEW** `src/lib/field/applyTemplate.ts`

- `applyPropertyTemplate(vistoriaId: string, templateId: string): Promise<{ ambientes: number; items: number }>`
  - Ler ambientes da vistoria; se `length > 0`, throw ou retornar erro tipado `ALREADY_HAS_ROOMS` (UI confirma reset só se implementarmos reset — **MVP: bloquear** e mostrar “já existem cômodos; apague-os ou use + Cômodo”).
  - Para cada room do template, em ordem:
    - `LocalAmbiente` com id `amb-local-${Date.now()}-${i}` (evitar colisão com delay mínimo ou counter)
    - Itens com id `item-local-${Date.now()}-${i}-${j}`, `status: "PENDENTE"`, descricao null
  - Enfileirar mutações **por ambiente e por item** (ver T-03)
  - Retornar contagens

- `createAmbienteWithDefaultItems(vistoriaId, nome, ordem?): Promise<LocalAmbiente>`
  - Cria 1 ambiente + itens de `defaultItemsForRoomName(nome)`
  - Usado pelo modal “+ Cômodo” existente

#### T-03: Normalizar mutações de create

**MODIFY** `src/lib/db/idb.ts` (se necessário alinhar actions)

- Garantir actions usadas:
  - `CREATE_AMBIENTE_LOCAL` payload: `{ ambiente: LocalAmbiente }`
  - `CREATE_ITEM_LOCAL` payload: `{ item: LocalItem }`
- Atualizar páginas que hoje usam `UPDATE_VISTORIA_STATUS` + `tipo: CREATE_*` para as actions tipadas (ambientes + item pages).

**MODIFY** `src/app/api/vistorias/[id]/route.ts` (PUT handler)

- Aceitar `CREATE_AMBIENTE_LOCAL` / `CREATE_ITEM_LOCAL` (e aliases antigos se existirem) criando registros no Prisma com os IDs enviados **ou** gerando UUID e…  
  - **Preferência MVP:** se IDs forem `*-local-*`, servidor gera UUID e… **problema de remap**.  
  - **Decisão prática Plan 01:**  
    - Offline: IDs locais no IDB.  
    - Sync: se PUT ainda não suporta create, **não bloquear field** — mutações ficam na fila; documentar gap se API não criar.  
    - Implementar create no PUT se ainda não existir: corpo com nome, ordem, vistoriaId; para item: nome, ambienteId.  
    - Se `ambienteId` ainda for local e não existir no server, **ordenar mutações** ambiente-before-items (já é FIFO da fila).

> Spike curto no início do Plan 01: ler o PUT atual e listar actions suportadas. Se create não existir, implementar mínimo no mesmo plan.

## Plan 02 — Navegação “próximo item”

### Objetivo

Após salvar um item na captura, o vistoriador cai no próximo pendente sem voltar à lista do cômodo.

### Tasks

#### T-05: Helper de ordem de itens

**NEW** `src/lib/field/itemNavigation.ts`

- `async function listOrderedItems(vistoriaId: string): Promise<Array<{ item: LocalItem; ambienteId: string; ambienteNome: string; ordemAmbiente: number }>>`
  - Ambientes da vistoria ordenados por `ordem`
  - Itens por ambiente (ordem de criação / `createdAt` se não houver campo ordem no item)
- `function findNextPending(currentItemId, list): next | null`
  - Preferir pendentes **depois** do atual no mesmo ambiente
  - Depois pendentes em ambientes seguintes
  - Depois pendentes **antes** (wrap) se política for “esvaziar fila”; **MVP: sem wrap** — se não houver próximo, `null`
- `function buildItemPath(vistoriaId, ambienteId, itemId): string`

#### T-06: Captura — salvar e avançar

**MODIFY** `src/app/field/vistorias/[id]/ambientes/[ambienteId]/itens/[itemId]/page.tsx`

- **Gate de foto (D-05):** `canSave` exige **≥ 1 foto** + estado (BOM/REGULAR/RUIM). Sem foto: botões de salvar desabilitados + mensagem curta (“Tire ao menos 1 foto — o relatório precisa de evidência visual”).
- Áudio **não** é obrigatório para salvar/avançar nesta onda.
- Após `handleSaveItem` bem-sucedido:
  - Resolver próximo pendente (excluindo o item recém-salvo, que já não é PENDENTE)
  - Se existe: `router.replace` (ou push) para o path do próximo
  - Se não: `router.push` lista do ambiente (comportamento atual) **ou** resumo se zero pendentes na vistoria
- UI:
  - Botão primário: **“Salvar e próximo”** (default)
  - Secundário texto/link: **“Salvar e voltar à lista”** (preserva escape hatch)
- Cabeçalho: indicador discreto `Item 3/8 neste cômodo` (opcional, se barato)
- Botões **Anterior / Próximo** no topo (navegam entre itens do ambiente mesmo se já vistoriados) — nice-to-have; se estourar tempo, só “Salvar e próximo”

#### T-07: Lista do ambiente — atalho “continuar”

**MODIFY** `src/app/field/vistorias/[id]/ambientes/[ambienteId]/page.tsx`

- Se existir item PENDENTE, botão sticky **“Continuar vistoria”** → abre o primeiro pendente
- Badge por item: PENDENTE / BOM / REGULAR / RUIM (já parcial — garantir contraste)

### Critérios de aceite — Plan 02

- [ ] Sem foto: não é possível salvar o item (estado sozinho não basta)
- [ ] Com ≥ 1 foto + estado: “Salvar e próximo” avança
- [ ] Salvar item 1 com pendentes restantes abre item 2 sem passar pela lista
- [ ] Último pendente do imóvel ao salvar volta à lista (ou resumo)
- [ ] “Salvar e voltar à lista” sempre disponível
- [ ] Funciona 100% offline

### Done = commit

`feat(field): auto-advance to next pending item after save`

## Plan 03 — Score de completude acionável

### Objetivo

No resumo, o vistoriador vê o que falta e toca para ir corrigir. **Finalize só libera** com zero itens PENDENTE **e** todo item com ≥ 1 foto (D-05 — relatório fotográfico exige evidência visual).

### Tasks

#### T-08: Cálculo de completude

**NEW** `src/lib/field/completionScore.ts`

- Input: checklist, ambientes, items, midias
- Output:

```ts
type CompletionIssue = {
  id: string;
  severity: "blocker" | "warn" | "info";
  label: string;
  href: string; // deep link field
};

type CompletionScore = {
  percent: number; // 0–100
  issues: CompletionIssue[];
  stats: {
    ambientes: number;
    itemsTotal: number;
    itemsDone: number;
    itemsWithoutPhoto: number;
    checklistDone: number;
    checklistTotal: number;
    midiasPendingSync: number;
  };
};
```

- Regras de percentagem (simples e documentadas; foto pesa de verdade):
  - 10% cadastro mínimo: tem ≥ 1 ambiente
  - 10% protocolo: checklistDone/checklistTotal * 10
  - 40% itens com status: itemsDone/itemsTotal * 40 (0 se total 0)
  - **40% cobertura foto:** itens com ≥1 foto / itemsTotal * 40
- Flag derivada: `canFinalize = ambientes ≥ 1 && itemsTotal ≥ 1 && itemsPending === 0 && itemsWithoutPhoto === 0`
- Issues (exemplos):
  - **blocker:** nenhum ambiente → `/field/vistorias/{id}/ambientes`
  - **blocker:** itens ainda PENDENTE → link para primeiro pendente ou ambientes
  - **blocker:** item sem foto (mesmo se status ≠ PENDENTE por dado legado) → link captura do item
  - warn: protocolo incompleto → `/ambientes` (seção checklist)
  - info: mídias pendentes de sync → `/field/sync`
  - info: item sem áudio (opcional; não bloqueia) → link captura

#### T-09: UI resumo

**MODIFY** `src/app/field/vistorias/[id]/resumo/page.tsx`

- Card “Completude” com Progress + %
- Lista de issues (máx. 8 visíveis + “e mais N”)
- Cada issue = `Link` com min-height 44px
- Manter card de sync e CTA “Finalizar Vistoria”
- **Disable finalize** se qualquer uma for verdadeira (D-05):
  - 0 ambientes
  - algum item ainda `PENDENTE`
  - algum item **sem ≥ 1 foto** (FOTO no IDB daquele item)
- Mensagem sob o botão quando bloqueado por foto:  
  *“Todo item precisa de ao menos 1 foto para constar no relatório.”*
- Áudio ausente **não** desabilita finalize

#### T-10: Espelhar % leve na home do cômodo (opcional no mesmo plan se couber)

**MODIFY** `ambientes/page.tsx` — reutilizar `computeCompletionScore` no header progress existente (hoje já tem fórmula ad-hoc `cadastroPct + checklist + items`). Substituir pela função única para **uma só verdade**.

### Critérios de aceite — Plan 03

- [ ] Vistoria vazia: % baixo + issue “Adicione cômodos ou aplique um modelo”
- [ ] Template aplicado, zero captura: blockers de itens pendentes **e** sem foto
- [ ] Toque em “X itens sem foto” navega para um item sem foto
- [ ] Finalize **bloqueado** se houver item sem foto (mesmo com status BOM/REGULAR/RUIM)
- [ ] Finalize **habilitado** só com: ≥1 ambiente, zero PENDENTE, todo item com ≥1 foto
- [ ] Item sem áudio não bloqueia finalize
- [ ] Offline

### Done = commit

`feat(field): actionable completion score on inspection summary`

## Validação manual (UAT mínimo da fase)

1. Login field → Nova vistoria local (endereço mínimo) → Ambientes → aplicar **Apto 2Q**.
2. Abrir primeiro cômodo → Continuar → capturar foto + marcar BOM → **Salvar e próximo** (repete 3 itens).
3. Voltar lista → ver badges atualizados.
4. Resumo → ver % e issues (protocolo, fotos faltantes).
5. Completar um cômodo inteiro (status + foto em cada item) → issues de foto somem.
6. Tentar finalizar com item BOM mas **sem foto** → botão bloqueado + mensagem de evidência visual.
7. Modo avião: aplicar template + capturar foto + avançar item (sem rede).
8. Online: sync não corrompe lista (se create no server existir; senão documentar “local-only creates” como gap conhecido).

## Riscos e mitigações

| Risco | Mitigação |
|-------|-----------|
| PUT não cria ambiente/item no server | Spike no dia 1 do Plan 01; create mínimo ou fila “best effort” + nota no SUMMARY |
| Template gera 40+ items e UI lenta | Virtualização não necessária no MVP; listas por cômodo já fragmentam |
| Usuário aplica template errado | Bloquear re-apply; “apagar todos os cômodos” **fora** desta onda (evitar delete em massa arriscado) |
| Conflito com vistorias seed do admin | Template só se 0 ambientes; seed com ambientes já feitos não mostra apply |
